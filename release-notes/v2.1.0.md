Release Notes — Version 2.1.0
Release Date: July 1, 2026
Type: Minor release
---
What's New
Cursor-Based Pagination for All List Endpoints
All endpoints that return lists now support cursor-based pagination. This
replaces the previous offset-based pagination and provides more reliable
results when data changes between requests.
Use the `next_cursor` value from any list response as the `cursor` parameter
in your next request to retrieve the following page.
For more information, see Pagination.
New `status` Filter on List Workflows
You can now filter the `/v2/workflows` endpoint by status. Pass
`?status=active`, `?status=inactive`, or `?status=draft` to retrieve only
workflows matching that status.
Improved Rate Limit Headers
Rate limit responses now include two additional headers:
`X-RateLimit-Remaining` — the number of requests remaining in the current window
`X-RateLimit-Reset` — the Unix timestamp when the rate limit window resets
---
Changes
Deprecation: Offset Pagination
Offset-based pagination using `?page=` and `?per_page=` parameters is now
deprecated. These parameters continue to work in v2 but will be removed in v3.
Migrate to cursor-based pagination before the v3 release.
---
Bug Fixes
Fixed an issue where deleting a workflow with active in-progress runs
returned a `500` error instead of completing the deletion correctly
Fixed an issue where the `updated_at` timestamp was not updated when
a workflow's status changed from `active` to `inactive`
Fixed an issue where API keys with Read Only scope could incorrectly
create workflows under certain conditions
---
Migration Guide
If you use the `?page=` or `?per_page=` parameters in your current
integration, update your code to use cursor-based pagination before the v3
release. See Pagination for the
updated implementation.
